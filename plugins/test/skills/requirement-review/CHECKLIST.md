# 需求评审基线

本文件提供 `requirement-review` 工作流在 `review` 阶段使用的 12 维度评审检查项和标准 QA Checklist。`SKILL.md` 只保留入口和执行闭环，逐项检查时再读取本文件。

## 术语说明

| 术语 | 含义 |
| --- | --- |
| TapSDK | 公司自研客户端 SDK，负责数据采集、推送和基础服务能力 |
| 建站器 | 内部运营活动页面搭建平台，活动页通过建站器发布和更新 |
| DE（数仓） | Data Engineering 数据工程团队，负责数据仓库和数据管道 |
| IEM | 内部事件管理平台，负责事件埋点定义和分析 |

## 需求质量评审维度（12 维度）

对 `requirement_understanding.md` 中的每个功能点和业务规则，从以下 12 个维度逐项评审。各维度统一使用 [TEMPLATES.md](TEMPLATES.md#review_checklistmd-模板) 中定义的表格格式输出。

### 1. 功能边界

**检查点**：
- 需求描述的功能范围是否有明确的内/外边界
- 是否存在模糊的「等」「等等」「类似」表述
- 新功能与现有功能的交互边界
- 功能是否覆盖完整 CRUD 链路（创建/查看/编辑/删除）

**问题模板**：
- 「{功能}的范围是否包含{子功能A}和{子功能B}？」
- 「{功能}是否仅限{条件}场景，还是所有场景通用？」
- 「{功能}与现有{模块}的关系是替换、扩展还是独立？」

### 2. 状态流转

**检查点**：
- 核心实体是否有完整的状态定义
- 状态间的转换规则是否明确（合法/非法迁移）
- 终态是否可逆
- 并发状态变更的处理规则
- 业务规则之间是否存在冲突
- 数据流转是否闭环

**问题模板**：
- 「{实体}有哪些状态？状态之间的转换条件是什么？」
- 「{实体}处于{状态A}时，能否直接转到{状态C}，还是必须经过{状态B}？」
- 「{实体}到达{终态}后是否可以回退？」
- 「两个操作同时修改{实体}状态时如何处理？」

### 3. 性能要求

**检查点**：
- 响应时间要求（页面加载、接口响应）
- 数据量级预估（列表上限、批量操作上限）
- 并发场景预估
- 分页/懒加载策略
- 是否需要缓存策略
- 性能风险评估（大数据量、高并发场景）

**问题模板**：
- 「{列表}最多展示多少条数据？是否需要分页？」
- 「{操作}的预期响应时间是多少？」
- 「{功能}是否需要支持批量操作？上限是多少？」

### 4. 权限控制

**检查点**：
- 涉及的用户角色及其权限差异
- 数据可见性规则（哪些角色能看到哪些数据）
- 操作权限（哪些角色能执行哪些操作）
- 未登录/游客的访问限制

**问题模板**：
- 「{功能}涉及哪些用户角色？各角色的权限差异是什么？」
- 「{数据}对{角色A}和{角色B}的可见范围是否不同？」
- 「未登录用户能否访问{功能}？」

### 5. 异常处理

**检查点**：
- 网络异常时的降级策略
- 操作失败后的回退/重试机制
- 数据一致性保障（如支付场景）
- 中断场景处理（页面刷新、切后台、超时）
- 回退/撤销方案是否考虑

**问题模板**：
- 「{操作}失败后，{实体}的状态如何处理？用户看到什么提示？」
- 「{操作}进行中用户刷新页面/切后台，回来后如何恢复？」
- 「{操作}超时后是自动重试还是需要用户手动操作？」

### 6. 数据约束

**检查点**：
- 输入字段的长度/格式/类型限制
- 唯一性约束（如用户名不可重复）
- 必填/选填规则
- 数据保留和清理策略
- 边界条件是否有定义（最大值、最小值、空值、特殊字符、超长文本）

**问题模板**：
- 「{字段}的长度限制是多少？是否允许特殊字符？」
- 「{字段}是否需要唯一？重复时如何提示？」
- 「{数据}的保留期限是多久？过期后如何处理？」

### 7. 依赖关系

**检查点**：
- 上下游系统依赖（第三方 API、内部服务）
- 多端一致性要求（Server/Android/iOS/Web/PC）
- 多端是否同步开发和上线？如果存在时间差，业务表现预期需要对齐
- 是否需要 DE（数仓）或 IEM 支持
- 确认有无新增数据埋点
- 是否涉及 TapSDK，变更是否影响 TapSDK
- 功能实现是否依赖宿主发版
- 是否有其他并行项目会改到同一块代码，导致相互干扰
- 是否需要新增数据表/字段
- 是否涉及数据迁移

**问题模板**：
- 「{功能}是否依赖{外部服务}？该服务不可用时如何降级？」
- 「多端（Android/iOS/Web）的行为是否完全一致？是否同步上线？」
- 「是否需要新增数据埋点？具体事件是什么？」
- 「是否有其他并行项目会改到同一块代码？」

### 8. 交互与 UI 规则

**检查点**：
- 用户操作流程和页面跳转路径
- 表单交互规则（实时校验/提交校验、联动、默认值）
- 加载状态和空状态展示
- 动效和手势交互（移动端）
- 提示信息（成功/失败/确认弹窗）
- 信息层级是否清晰
- 如有设计稿，对照设计稿验证交互逻辑和状态覆盖

**问题模板**：
- 「{操作}完成后跳转到哪个页面？是否有成功提示？」
- 「{表单}中{字段A}变化时，{字段B}是否联动更新？」
- 「{列表}为空时展示什么？加载中展示什么？」
- 「{操作}是否需要二次确认弹窗？」

### 9. 安全与合规

**条件触发**：仅当需求涉及用户输入、权限、支付或敏感数据时逐项检查，否则整体标记为 `[不适用]`。

**检查点**：
- 新增/修改的用户输入是否有校验方案（防 XSS、注入）
- 权限控制变更是否考虑越权场景（水平越权、垂直越权）
- 涉及金额计算时精度和防篡改方案是否明确
- 敏感数据的存储/传输/展示是否有脱敏方案
- 日志中的敏感信息脱敏规则
- 操作审计记录需求
- 隐私法规合规（如需用户授权才能采集的数据）
- 防刷和风控策略

**问题模板**：
- 「{敏感数据}在日志和展示中是否需要脱敏？脱敏规则是什么？」
- 「{操作}是否需要记录审计日志？记录哪些字段？」
- 「{功能}涉及的用户数据采集是否需要用户授权？」
- 「{接口}是否需要防刷限制？频率是多少？」

### 10. 兼容性

**检查点**：
- 浏览器版本支持范围（Web 端）
- 操作系统最低版本要求（移动端）
- 屏幕尺寸适配策略（响应式/自适应/固定宽度）
- 旧版本客户端的降级或强制升级策略
- 新老版本兼容方案如何设计
- 国际化/本地化需求

**问题模板**：
- 「需要支持哪些浏览器？最低版本是什么？」
- 「最低支持的 Android/iOS 版本是多少？」
- 「小屏幕（如 320px 宽）下{组件}如何展示？」
- 「旧版本客户端访问新功能时如何处理？强制升级还是降级展示？」

### 11. 可测试性与验收标准

**检查点**：
- 每个功能点的验收标准是否明确且可验证
- 是否需要特殊的测试数据或测试账号
- 是否有需要模拟的外部依赖（Mock 服务）
- 验收由谁执行、在什么环境执行
- 是否需要特殊测试手段（自动化/性能/安全/兼容性测试）
- 灰度/AB 测试需求
- 变更影响的回归范围是否可评估

**问题模板**：
- 「{功能点}的验收标准是什么？如何判断"做完了"？」
- 「测试{功能}需要准备哪些测试数据？是否需要特殊账号？」
- 「{外部依赖}在测试环境中如何处理？有 Mock 服务吗？」
- 「{功能}是否需要灰度发布？灰度策略是什么？」

### 12. 影响范围

**检查点**：
- 变更涉及的模块和功能（直接影响 + 间接影响）
- 上下游系统是否需要同步调整
- 数据库字段变更是否影响报表、数据分析
- 是否需要数据迁移或回填
- 共用组件/SDK 的修改是否影响其他业务
- 是否影响现有接口（向前/向后兼容）
- 是否有技术债务风险

**问题模板**：
- 「修改{模块}会影响哪些下游模块？{列表}是否都在本次范围内？」
- 「{字段}变更后，报表/数据分析是否需要同步调整？」
- 「{共用组件}的修改是否会影响其他业务线？」
- 「是否需要数据迁移？历史数据如何处理？」

## 标准 QA Checklist

> 以下为项目级检查项，无法归入 12 维度的功能点级评审。与维度评审互补使用。

### 工作流程

- 飞书会议、妙记是否开启？

### 业务细节

- 需求是否涉及其它业务线（Android、iOS、PC）
- 与买断制相关的改动时，需要考虑自测用户 0.01 元购买的场景

### 测试环境

- 核心功能（如支付、搜索）在研发环境能联调吗？会不会因为环境没配好而卡住？
- 复杂的测试数据（如游戏战绩）谁来造？怎么造？
- 如果测试环境确实不支持（如涉及到真实金钱流转），是否准备好了 Mock（模拟）方案？
- 明确"提测"的时间点。是当天下班前，还是次日一早？（建站器参与的项目尤其注意）

### 项目发布相关

- 梳理好：测试环境 -> 预发布 -> 正式环境 的完整时间路线图以及各个职能需要执行的动作

## 使用建议

- review 阶段先回读 `requirement_understanding.md`，再按 12 维度逐项检查
- 根据需求类型（understand 阶段的分类结果）选择适用的维度，跳过明显不相关的维度并标记为 `[不适用]`
- **维度优先级**：功能边界 → 状态流转 → 交互与UI规则 → 依赖关系 → 异常处理 → 数据约束 → 权限控制 → 安全与合规 → 兼容性 → 可测试性 → 影响范围 → 性能要求
- 评审发现的问题按照 TEMPLATES.md 中的问题分配规则分配给对应职能
