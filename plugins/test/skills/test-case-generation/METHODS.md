# 测试设计方法论详细指南

本文件是方法论的详细参考，供子 Agent 或主 Agent 按需 Read。SKILL.md 中的「方法选择指引」表提供快速决策入口。

## 等价类划分（Equivalence Partitioning）

将输入/条件划分为若干等价类，从每类中选取代表值设计用例。

- **适用场景**：输入域可划分（文本框、下拉选择、数值范围、用户角色/权限等级）
- **操作方法**：
  - 识别所有输入/条件的有效等价类和无效等价类
  - 有效类：符合规则的各类典型值
  - 无效类：超出规则的各类异常值（空值、非法字符、超长、类型错误等）
  - 每个等价类至少一条用例
- **典型优先级**：有效类 P0-P1，无效类 P1-P2
- **业务示例**：用户昵称输入框（限制 1-20 个字符）→ 有效类：普通中文名"张三"、英文名"Alice"、混合"用户A1"；无效类：空字符串、纯空格、含特殊符号"<script>"、超过 20 字符的长字符串

## 边界值分析（Boundary Value Analysis）

在等价类的边界点及其邻近值设计用例，边界处最易出错。

- **适用场景**：数值范围、长度限制、日期/时间区间、分页/数量上下限、状态阈值
- **操作方法**：
  - 取边界值：min、min+1、max-1、max
  - 取越界值：min-1、max+1
  - 对日期/时间取起止边界（活动开始前一秒、结束后一秒等）
  - 对字符长度取 0、1、max-1、max、max+1
- **典型优先级**：常见边界 P1，极端边界 P3
- **业务示例**：商品数量选择（范围 1-999）→ 边界值：1、2、998、999；越界值：0、1000、-1；优惠券有效期 → 生效前 1 秒、失效后 1 秒

## 场景法（Scenario Testing）

基于用户实际使用的业务流程，设计端到端场景用例。

- **适用场景**：多步骤业务流程、用户旅程、涉及多角色协作的功能
- **操作方法**：
  - 梳理主流程（Happy Path）和各分支流程
  - 每条场景覆盖一个完整的业务路径
  - 包含正向主流程、备选流程、异常中断与恢复流程
- **典型优先级**：主流程 P0，备选流程 P1，异常中断 P2
- **业务示例**：用户购买会员 → 主流程：选择套餐 → 确认支付 → 支付成功 → 会员生效；备选：使用优惠券支付；异常：支付中途断网 → 返回后重试 → 幂等校验

## 错误推测法（Error Guessing）

基于经验推测易出错的场景，补充其他方法难以覆盖的薄弱点。

- **适用场景**：所有功能，尤其是网络异常、并发操作、权限越界、数据一致性等
- **操作方法**：
  - 网络异常（断网、弱网、超时）下的系统行为
  - 并发/重复操作（快速连续点击、多端同时操作）
  - 非常规输入（特殊字符、SQL 注入、XSS、超大文件）
  - 中断场景（操作中途刷新、关闭、切后台）
  - 缓存/状态残留导致的异常
- **典型优先级**：P2-P3

## 判定表法（Decision Table Testing）

当功能涉及多个条件组合且输出随组合变化时，用判定表穷举条件组合。

- **适用场景**：多条件联合判断（权限 + 状态 + 角色组合、多开关联动、复杂业务规则）
- **操作方法**：
  - 列出所有条件及其取值
  - 构建条件组合矩阵
  - 合并结果相同的组合以精简用例
  - 每个有效组合至少一条用例
- **典型优先级**：核心组合 P1，次要组合 P2
- **业务示例**：内容发布权限 → 条件：用户角色（普通/VIP/管理员）× 内容状态（草稿/审核中/已发布）× 操作（编辑/删除/置顶），构建组合矩阵后合并结果相同的行

## 状态迁移法（State Transition Testing）

针对具有明确状态流转的功能，验证各状态间的转换是否正确。

- **适用场景**：订单/工单状态流转、任务生命周期、审批流程、开关/模式切换
- **操作方法**：
  - 绘制状态迁移图（状态 + 事件 → 新状态）
  - 覆盖所有合法迁移路径
  - 验证非法迁移是否被正确拒绝
  - 关注状态回退、循环、终态不可变
- **典型优先级**：合法迁移 P0-P1，非法迁移 P1-P2
- **业务示例**：工单状态流转 → 待办 → 研发中 → 已修复 → 完成；非法迁移：待办不能直接跳到完成；回退：已修复 → 重新打开 → 研发中

## 探索性测试法（Exploratory Testing）

基于测试章程（Charter）和时间盒（Session），在有限范围内自由探索产品行为，发现脚本化用例难以覆盖的缺陷。

- **适用场景**：新功能上线前的风险探索、复杂交互/用户体验评估、需求描述模糊但需要快速发现问题的区域、其他方法覆盖后仍有"直觉性风险"的模块
- **与其他方法的区别**：等价类/边界值/场景法等是"验证已知行为是否正确"，探索性测试是"发现未知问题"。两者互补，不替代
- **操作方法**：
  - 编写测试章程（Charter）：明确探索目标、关注区域和判定标准
  - 章程格式：`探索 {目标区域}，使用 {方法/策略}，发现 {风险类型}`
  - 为每个章程定义探索要点（Exploration Points）：列出需要关注的交互路径、数据组合或环境条件
  - 为每个探索要点定义判定标准（Oracle）：如何判断行为是否异常（一致性、可用性、性能感知、数据完整性等）
  - 建议结合常见探索启发式（Heuristics）：
    - HICCUPPS：历史(History)、一致性(Consistency)、可比性(Comparability)、声明(Claims)、用户期望(User expectations)、产品规律(Product)、标准(Standards)
    - 漫游测试法：地标游(Landmark)、极端游(Supermodel)、深巷游(Alleyway)、通宵游(All-Nighter)
- **典型优先级**：P1-P2（核心区域的探索 P1，补充性探索 P2）
- **用例格式说明**：探索性测试用例复用标准 JSON 结构，但语义不同：
  - `steps[].action`：填写探索要点/操作方向（非精确步骤），如"在弱网环境下反复切换支付方式"
  - `steps[].expected`：填写判定标准/Oracle（非精确预期），如"切换过程应无数据丢失，金额始终一致"
  - `preconditions`：包含章程（Charter）描述
  - `title`：以探索目标为主语，如"探索多端同时编辑时的数据同步行为"
- **业务示例**：电商下单功能 → Charter：`探索 支付中断与恢复流程，使用 多种中断方式（断网/切后台/杀进程），发现 数据一致性和幂等性问题`；探索要点：支付中断后返回是否保留订单状态、重复提交是否产生重复扣款、不同中断时机（选择支付方式时/输入密码时/等待回调时）的行为差异
