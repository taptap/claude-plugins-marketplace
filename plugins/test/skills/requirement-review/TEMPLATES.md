# 需求评审模板集

本文件包含 `requirement-review` 工作流各阶段使用的中间文件和输出模板。

**目录**（按阶段使用顺序）：

1. `requirement_understanding.md` 模板 — understand 阶段输出
2. `review_checklist.md` 模板 — review 阶段输出
3. `report.md` 模板 — output 阶段输出（含 7 章结构定义）
4. 问题分配规则 — review/output 阶段使用
5. 优先级定义 — review/output 阶段使用

---

## report.md 模板

> `output` 阶段使用本模板组织评审报告，写入 `report.md`。飞书文档导出由平台自动处理。
> 以下给出每章的**格式示例**，LLM 必须严格遵循示例中的格式和标点风格。

### 完整报告示例骨架

```markdown
# {需求名称} — 需求评审报告

## 1. 评审结论

A-1（FB2）: {问题简述} | {所需行动}
A-2（ST1）: {问题简述} | {所需行动}
A-3（EH3）: {问题简述} | {所需行动}
元数据：
需求名称：XXX

需求链接：https://...

F2: {模块} | {功能点描述} | 显式
```

### 各章节格式规则

**第 1 章「评审结论」**

- 第一行格式固定：`就绪度：{标记} {判定}`，标记三选一：`[OK]` / `[!]` / `[X]`
- 紧跟一段自然语言总结（1-2 句），概括核心问题域，不泛泛而谈
- 就绪条件列表：每行格式 `A-N（来源编号）: 问题简述 | 所需行动`
- 元数据：每行一个 `key：value`，不使用表格

**第 2 章「需求理解」**

- 需求目标：以 `需求目标：` 开头，1-2 句话。不复述用户场景
- 功能点清单：以 `功能点清单（共 N 个）：` 开头
- 每个功能点一行，格式 `F{N}: {模块} | {描述} | 显式/隐式`
- 功能点必须细粒度（每个可独立测试），不能笼统分组
- 相同模块的功能点连续排列，通过模块名自然分组

**第 3 章「各职能待确认问题列表」**

报告的核心价值章节。按 PM / Dev / QA / Design 分组。

格式规则：
- 每组以 `### {角色}（{中文名}）` 为标题
- 每个问题一行，格式 `{角色}-{N}: {问题描述} | {优先级} | {来源编号}-{维度名} | {建议确认时间}`
- 问题描述必须具体、可回答，包含足够上下文（如引用文档中的矛盾文案原文）
- 阻断项排在每组最前面
- 来源编号关联 review_checklist.md 中的检查项编号（如 FB2、ST1、CL10）

**第 4 章「风险项摘要」**

风险筛选标准——满足以下任一条件的评审发现应提升为风险项：
- 性能要求维度（PF）中预估数据量级过大或响应时间有风险的项
- 异常处理维度（EH）中影响核心流程的项
- 依赖关系维度（DP）中存在外部不可控因素的项
- 影响范围维度（IS）中涉及跨模块或跨系统变更的项
- 时间依赖或环境依赖可能导致延期的项

格式：每行 `R-{N}: {风险描述} | {优先级} | {影响范围} | {建议措施}`
无风险时标注"未发现重大风险"。

**第 5 章「评审范围声明」**

纯文本列表，标注信息边界。必须包含：评审文档列表、设计稿获取状态、技术方案获取状态、已知信息缺失。

**第 7 章「评审完整性声明」**

纯文本统计列表（不使用表格），聚焦覆盖率和完整性验证。不重复第 3 章已展示的维度分布数字。必须包含：功能点覆盖率、评审项目总计、问题发现分布、阻断项比例及合理性判断、风险项概要、未覆盖内容。

### 关键约束

- 整份报告使用纯文本和分隔线排版，**不使用表格**（飞书导入后表格渲染不稳定）
- 所有问题详情归入第 3 章
- 第 3 章每个问题独占一行，格式统一为 `编号: 问题 | 优先级 | 来源 | 确认时间`
- 阻断项通常不超过总问题数的 20%；若超过，在第 1 章评审结论中说明需求成熟度不足
- 功能点编号（F1/F2...）在报告全文中保持一致，第 3 章问题可通过功能点编号交叉引用

## 问题分配规则

将评审发现的问题按以下规则分配给对应职能。12 维度 → 职能映射：

| 评审维度 | 主分配 | 副分配 |
| --- | --- | --- |
| 功能边界 | PM | — |
| 状态流转 | PM | Dev |
| 性能要求 | Dev | — |
| 权限控制 | Dev | QA |
| 异常处理 | Dev | QA |
| 数据约束 | Dev | QA |
| 依赖关系 | Dev | PM |
| 交互与 UI 规则 | Design | PM |
| 安全与合规 | Dev | — |
| 兼容性 | Dev | QA |
| 可测试性与验收标准 | QA | PM |
| 影响范围 | PM | Dev |

补充分配规则（适用于跨维度问题）：

| 问题来源 | 分配给 |
| --- | --- |
| 跨端一致性、多端同步 | PM + Dev（联合确认） |
| 测试环境、测试数据、提测时间、Mock 方案 | QA（测试） |
| 空状态设计、加载状态、错误提示 | Design（设计） |
| 文档-设计稿差异（功能覆盖） | PM（产品） |
| 文档-设计稿差异（交互行为/UI 状态） | Design（设计） |
| 文档-设计稿差异（数据字段） | PM + Dev（联合确认） |

## 优先级定义

| 优先级 | 含义 | 建议确认时间 |
| --- | --- | --- |
| 阻断 | 不确认无法进入开发，严重影响需求范围 | 评审前或评审会上必须确认 |
| 高 | 影响核心功能或有较大风险 | 评审会上确认 |
| 中 | 影响部分功能或需要补充说明 | 开发前确认 |
| 低 | 优化建议或细节问题 | 开发过程中确认即可 |
