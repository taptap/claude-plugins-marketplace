## 📋 缺陷要素规范

| 缺陷要素 | 要求 | 备注/示例 |
|---------|------|----------|
| **标题** | 标题清晰，明确问题；包含受影响的端/功能模块 | `[游戏详情页]点击在线玩，偶现云游戏启动失败` |
| **描述** | 详细描述（见下方详细说明） | 包含前置条件、复现步骤、实际结果、预期结果 |
| **优先级** | 根据业务需求和影响范围定级 | P0/P1/P2/P3 |
| **版本** | 影响版本和修复版本 | 见版本规范说明 |

---

## 📝 描述字段规范

描述要详细，包括但不限于：

### 1. 前置条件
- 测试环境
- 测试账号
- 测试数据等

### 2. 复现步骤
- 具体操作步骤
- 附件截图/录屏
- **注明复现频次**：必现、偶现（标明偶现频次/偶现次数）

### 3. 实际结果
- 客观、准确地描述发生了什么
- ❌ 避免使用"不工作"、"坏了"等模糊词汇

### 4. 预期结果
- 明确描述在执行上述步骤后，系统本应如何表现

### 5. 补充说明
- 涉及接口传参需附上**抓包信息**
- 埋点问题需要附上**日志**

## 🏷️ Bug 标题前缀定义

| 标题前缀 | 说明 | 使用场景 |
|---------|------|---------|
| `[线上]` | 各端线上 bug | 线上环境发现的问题 |
| `[回归]` | 回归阶段提出的 bug | 回归测试阶段发现 |
| `[日志]` | 日志埋点相关的 bug | 埋点漏报、错报等 |
| `[兼容性]` | 设备兼容性 bug | 特定设备/系统版本问题 |
| `[偶现]` | 偶现类 bug | 非必现问题 |
| `[AutoTest]` | UI 自动化发现的功能缺陷 | 自动化测试发现 |

**标题示例**：
- `[线上][游戏详情页] iOS 端点击下载按钮无响应`
- `[偶现][动态] Android 端发布动态偶现失败（3/10）`
- `[兼容性][播放器] 华为 Mate60 视频播放黑屏`

## 🔴 优先级定义

### 总体原则

| 优先级 | 定义 | 解决要求 |
|--------|------|---------|
| **P0** | 导致系统崩溃、数据丢失或功能无法使用 | 需要**立即解决** |
| **P1** | 影响关键功能，但有临时解决方案 | 应**尽快解决** |
| **P2** | 影响次要功能，不会导致系统崩溃 | 可以在**后续版本**中修复 |
| **P3** | 对用户体验或功能影响较小 | 可以安排在**未来版本**中解决 |

---

### 客户端 / SDK 优先级

| 优先级 | 定义 | 解决时间要求 |
|--------|------|-------------|
| **P0** | • 核心功能完全不可用<br>• 必现 Crash<br>• 广告等直接影响收入的埋点漏报、错报 | 第一时间响应，**24小时内**解决<br>解决方案：紧急发包 |
| **P1** | • 核心功能异常，不阻塞用户操作，但对用户体验或业务流程有重大影响<br>• 个人主页数据异常<br>• 其他埋点问题 | **2个工作日内**解决<br>解决方案：插件发版 or 随最近版本发布 |
| **P2** | • 非核心功能异常，不阻塞用户操作<br>• 不影响用户使用或可自行恢复使用的功能<br>• 可复现的兼容性问题 | **10个工作日内**解决<br>如需遗留需和产品确认并更新工单截止时间 |
| **P3** | • 对系统功能影响较小或仅影响界面美观<br>• 偶现/不可复现的兼容性问题 | 与产品研发讨论是否修复<br>以月度为单位进行清理 |

---

### Web 端优先级

| 优先级 | 定义 | 解决时间要求 |
|--------|------|-------------|
| **P0** | • 核心功能完全不可用<br>• 登录授权失败<br>• 核心页面无法打开 | 第一时间响应，**1小时内**恢复正常（包括撤版） |
| **P1** | • 核心功能异常，不阻塞用户操作<br>• 论坛页图片加载异常 | **2个工作日内**解决 |
| **P2** | • 非核心功能异常<br>• 偶现的功能缺陷<br>• 不影响用户使用或可自行恢复的功能问题 | **10个工作日内**解决<br>如需遗留需和产品确认并更新工单截止时间 |
| **P3** | • 对系统功能影响较小或仅影响界面美观<br>• 偶现/不可复现的兼容性问题 | 与产品研发讨论是否修复<br>以月度为单位进行清理 |

---

### 服务端优先级

| 优先级 | 定义 | 解决时间要求 |
|--------|------|-------------|
| **P0** | • 核心功能完全不可用<br>• 用户无法登录<br>• 游戏无法下载 | 第一时间响应，**1小时内**恢复正常（包括撤版） |
| **P1** | • 核心功能异常，不阻塞用户操作<br>• 游戏详情页无法看到评价列表 | **2个工作日内**解决 |
| **P2** | • 非核心功能异常，不阻塞用户操作 | **10个工作日内**解决<br>如需遗留需和产品确认并更新工单截止时间 |

## 🔵 严重程度定义

> **严重程度 = 技术破坏力**（这个 Bug 对系统造成了多大伤害？）
> **优先级 = 商业紧迫性**（我们需要多快修复它？）

| 严重程度 | 定义 | 特征 | 示例 |
|---------|------|------|------|
| **致命** | 系统彻底瘫痪，无法继续测试或使用 | 系统崩溃、死机、关键数据丢失、主要功能完全失效且无替代方案 | 登录按钮点击无反应；支付环节导致 App 闪退；数据库被清空 |
| **严重** | 核心功能受损，但系统未完全瘫痪 | 某个重要模块不可用，或功能逻辑有重大错误，但存在复杂的替代方案 | 无法导出报表（但可以预览）；搜索功能无法按日期排序（但能按名称排序） |
| **一般** | 功能受损，但不影响核心业务流程 | 功能未按预期运行，系统报错信息不准确，或特定边界条件下的小错误 | 错误提示语拼写错误但能看懂；点击"取消"按钮没有反应 |
| **轻微** | 主要是用户体验或界面问题，不影响功能使用 | UI/UX 问题，文案错误，布局微调 | 字体颜色不统一；按钮稍微对齐不整齐；免责声明中有错别字 |

### 严重程度 vs 优先级 对比

| 场景 | 严重程度 | 优先级 | 说明 |
|------|---------|--------|------|
| 登录功能崩溃 | 致命 | P0 | 高严重 + 高优先 |
| CEO 个人主页显示错误 | 轻微 | P0 | 低严重 + 高优先（商业原因） |
| 极少用户遇到的崩溃 | 致命 | P2 | 高严重 + 低优先（影响范围小） |
| 首页 Logo 颜色偏差 | 轻微 | P3 | 低严重 + 低优先 |

### 谁来决定？

| 指标 | 决定者 | 原因 |
|------|--------|------|
| **严重程度** | QA（测试工程师） | QA 最清楚 Bug 对系统的技术破坏力 |
| **优先级** | 产品经理 (PM) | PM 需要根据上线时间、客户需求和开发资源来拍板 |

## 📌 版本规范

### 基本规则
- **影响版本**：当前发布版本
- **修复版本**：当前发布版本

### 遗留 Bug 处理
- 修复的版本顺移到下一个版本
- 影响的版本为当前版本和下一个版本

**示例**：
- Bug 在 V1.0 版本中发现 → 影响版本: V1.0, 修复版本: V1.0
- Bug 在 V1.0 不修复 → 影响版本: V1.0, V2.0, 修复版本: V2.0

## 🔄 线上偶现类 Bug 处理原则

| 处理环节 | 要求 |
|---------|------|
| **优先级确定** | 根据基础 bug 优先级结合业务优先级组成最终优先级，按能复现调整优先级时需更新截止日期 |
| **追踪记录** | 接手 bug 后补充对应线索信息，同时跟研发对齐如何定位及解决，在工单留档并更新截止日期 |
| **响应时间** | 工单响应时间**不超过 2 天** |
| **关闭条件** | 若该 bug 经过定位以及用户侧 **3 个月内不再新增反馈**，则跟对应产品对齐后取消（cancel） |
