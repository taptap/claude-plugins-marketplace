# 需求评审基线

本文件提供 `requirement-review` 工作流在 `review` 阶段的检查项和评审策略。

12 维度检查点和问题模板见 [共享维度框架](../_shared/REQUIREMENT_DIMENSIONS.md)。

## 术语说明

| 术语 | 含义 |
| --- | --- |
| TapSDK | 公司自研客户端 SDK，负责数据采集、推送和基础服务能力 |
| 建站器 | 内部运营活动页面搭建平台，活动页通过建站器发布和更新 |

## 评审策略

### 维度优先级

功能边界 → 状态流转 → 交互与UI规则 → 依赖关系 → 异常处理 → 数据约束 → 权限控制 → 安全与合规 → 兼容性 → 可测试性 → 影响范围 → 性能要求

### 使用说明

- review 阶段先回读 `requirement_understanding.md`，再按 12 维度逐项检查
- 根据需求类型（understand 阶段的分类结果）选择适用的维度，跳过明显不相关的维度并标记为 `[不适用]`
- 评审发现的问题按照 TEMPLATES.md 中的问题分配规则分配给对应职能
- 各维度统一使用 [TEMPLATES.md](TEMPLATES.md#review_checklistmd-模板) 中定义的表格格式输出

### Go/No-Go 判定标准

- **阻断项**：影响功能可行性或存在重大风险的问题，评审前必须确认
- **关注项**：后续开发/测试中跟进即可
- 阻断项通常不超过总问题数的 20%；若超过须在评审结论中说明

### 问题职能分配

评审发现的问题按 PM/Dev/QA/Design 分组，使用统一的优先级定义（阻断/高/中/低）。

## 标准 QA Checklist

> 以下为项目级检查项，无法归入 12 维度的功能点级评审。与维度评审互补使用。

### 工作流程

- 飞书会议、妙记是否开启？

### 业务细节

- 需求是否涉及其它业务线（Android、iOS、PC）
- 与买断制相关的改动时，需要考虑自测用户 0.01 元购买的场景

### 测试环境

- 核心功能（如支付、搜索）在研发环境能联调吗？会不会因为环境没配好而卡住？
- 复杂的测试数据（如游戏战绩）谁来造？怎么造？
- 如果测试环境确实不支持（如涉及到真实金钱流转），是否准备好了 Mock（模拟）方案？
- 明确"提测"的时间点。是当天下班前，还是次日一早？（建站器参与的项目尤其注意）

### 项目发布相关

- 梳理好：测试环境 -> 预发布 -> 正式环境 的完整时间路线图以及各个职能需要执行的动作
