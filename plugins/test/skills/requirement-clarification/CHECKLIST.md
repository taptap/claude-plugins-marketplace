# 澄清维度清单

本文件提供 `requirement-clarification` 在 `clarify` 阶段使用的 12 个维度检查项。每个维度列出典型检查点和问题模板。

术语定义见 [CONVENTIONS.md](../../CONVENTIONS.md#术语表)。

## 1. 功能边界

**检查点**：
- 需求描述的功能范围是否有明确的内/外边界
- 是否存在模糊的「等」「等等」「类似」表述
- 新功能与现有功能的交互边界

**问题模板**：
- 「{功能}的范围是否包含{子功能A}和{子功能B}？」
- 「{功能}是否仅限{条件}场景，还是所有场景通用？」
- 「{功能}与现有{模块}的关系是替换、扩展还是独立？」

## 2. 状态流转

**检查点**：
- 核心实体是否有完整的状态定义
- 状态间的转换规则是否明确（合法/非法迁移）
- 终态是否可逆
- 并发状态变更的处理规则

**问题模板**：
- 「{实体}有哪些状态？状态之间的转换条件是什么？」
- 「{实体}处于{状态A}时，能否直接转到{状态C}，还是必须经过{状态B}？」
- 「{实体}到达{终态}后是否可以回退？」
- 「两个操作同时修改{实体}状态时如何处理？」

## 3. 性能要求

**检查点**：
- 响应时间要求（页面加载、接口响应）
- 数据量级预估（列表上限、批量操作上限）
- 并发场景预估
- 分页/懒加载策略

**问题模板**：
- 「{列表}最多展示多少条数据？是否需要分页？」
- 「{操作}的预期响应时间是多少？」
- 「{功能}是否需要支持批量操作？上限是多少？」

## 4. 权限控制

**检查点**：
- 涉及的用户角色及其权限差异
- 数据可见性规则（哪些角色能看到哪些数据）
- 操作权限（哪些角色能执行哪些操作）
- 未登录/游客的访问限制

**问题模板**：
- 「{功能}涉及哪些用户角色？各角色的权限差异是什么？」
- 「{数据}对{角色A}和{角色B}的可见范围是否不同？」
- 「未登录用户能否访问{功能}？」

## 5. 异常处理

**检查点**：
- 网络异常时的降级策略
- 操作失败后的回退/重试机制
- 数据一致性保障（如支付场景）
- 中断场景处理（页面刷新、切后台、超时）

**问题模板**：
- 「{操作}失败后，{实体}的状态如何处理？用户看到什么提示？」
- 「{操作}进行中用户刷新页面/切后台，回来后如何恢复？」
- 「{操作}超时后是自动重试还是需要用户手动操作？」

## 6. 数据约束

**检查点**：
- 输入字段的长度/格式/类型限制
- 唯一性约束（如用户名不可重复）
- 必填/选填规则
- 数据保留和清理策略

**问题模板**：
- 「{字段}的长度限制是多少？是否允许特殊字符？」
- 「{字段}是否需要唯一？重复时如何提示？」
- 「{数据}的保留期限是多久？过期后如何处理？」

## 7. 依赖关系

**检查点**：
- 上下游系统依赖（第三方 API、内部服务）
- 多端一致性要求（Server/Android/iOS/Web/PC）
- **前后端 API 契约**（当 `platform_scope.coordination_needed` 为 true 时必查）：
  - 新增接口的 method、path、request/response 字段定义
  - 现有接口是否需要新增字段或修改行为
  - 错误码定义和异常响应格式
  - 接口版本兼容策略（新旧客户端共存期间）
- 是否需要 DE/IEM 支持（数据埋点）
- 是否涉及 TapSDK
- 发版依赖（是否依赖宿主发版）
- 并行项目冲突风险

**问题模板**：
- 「{功能}是否依赖{外部服务}？该服务不可用时如何降级？」
- 「多端（Android/iOS/Web）的行为是否完全一致？是否同步上线？」
- 「{功能}需要新增 API 接口吗？请求和响应的字段有哪些？」
- 「{现有接口}是否需要新增字段？旧版客户端如何兼容？」
- 「是否需要新增数据埋点？具体事件是什么？」
- 「是否有其他并行项目会改到同一块代码？」

## 8. 交互与 UI 规则

**检查点**：
- 用户操作流程和页面跳转路径
- 表单交互规则（实时校验/提交校验、联动、默认值）
- 加载状态和空状态展示
- 动效和手势交互（移动端）
- 提示信息（成功/失败/确认弹窗）

**问题模板**：
- 「{操作}完成后跳转到哪个页面？是否有成功提示？」
- 「{表单}中{字段A}变化时，{字段B}是否联动更新？」
- 「{列表}为空时展示什么？加载中展示什么？」
- 「{操作}是否需要二次确认弹窗？」
- 「{列表项}支持左滑删除 / 长按拖拽排序吗？」

## 9. 安全与合规

**检查点**：
- 敏感数据的传输和存储加密方式
- 日志中的敏感信息脱敏规则
- 操作审计记录需求
- 隐私法规合规（如需用户授权才能采集的数据）
- 防刷和风控策略

**问题模板**：
- 「{敏感数据}在日志和展示中是否需要脱敏？脱敏规则是什么？」
- 「{操作}是否需要记录审计日志？记录哪些字段？」
- 「{功能}涉及的用户数据采集是否需要用户授权？」
- 「{接口}是否需要防刷限制？频率是多少？」

## 10. 兼容性

**检查点**：
- 浏览器版本支持范围（Web 端）
- 操作系统最低版本要求（移动端）
- 屏幕尺寸适配策略（响应式/自适应/固定宽度）
- 旧版本客户端的降级或强制升级策略
- 国际化/本地化需求

**问题模板**：
- 「需要支持哪些浏览器？最低版本是什么？」
- 「最低支持的 Android/iOS 版本是多少？」
- 「小屏幕（如 320px 宽）下{组件}如何展示？」
- 「旧版本客户端访问新功能时如何处理？强制升级还是降级展示？」
- 「是否需要支持多语言？首批支持哪些语言？」

## 11. 可测试性与验收标准

**检查点**：
- 每个功能点的验收标准是否明确且可验证
- 是否需要特殊的测试数据或测试账号
- 是否有需要模拟的外部依赖（Mock 服务）
- 验收由谁执行、在什么环境执行
- 灰度/AB 测试需求

**问题模板**：
- 「{功能点}的验收标准是什么？如何判断"做完了"？」
- 「测试{功能}需要准备哪些测试数据？是否需要特殊账号？」
- 「{外部依赖}在测试环境中如何处理？有 Mock 服务吗？」
- 「{功能}是否需要灰度发布？灰度策略是什么？」

## 12. 影响范围

**检查点**：
- 变更涉及的模块和功能（直接影响 + 间接影响）
- 上下游系统是否需要同步调整
- 数据库字段变更是否影响报表、数据分析
- 是否需要数据迁移或回填
- 共用组件/SDK 的修改是否影响其他业务

**问题模板**：
- 「修改{模块}会影响哪些下游模块？{列表}是否都在本次范围内？」
- 「{字段}变更后，报表/数据分析是否需要同步调整？」
- 「{共用组件}的修改是否会影响其他业务线？」
- 「是否需要数据迁移？历史数据如何处理？」

## 使用建议

- clarify 阶段先提炼功能点，再按功能点 × 维度矩阵逐项检查
- 文档已明确说明的标记 `source: "document"`，不向人提问
- 需要人工确认的按优先级分组批量提问，每次 3-5 个
- **维度优先级**：功能边界 + 平台范围 → 交互与 UI 规则 → 依赖关系（含 API 契约） → 状态流转 → 验收标准 → 异常处理 → 影响范围 → 其他
- 对用户不关心的维度提出默认假设，标记 `source: "assumption"` 让用户确认
- 探索模式下首轮重点在功能边界、平台范围和设计意图探查，避免过早深挖细节
