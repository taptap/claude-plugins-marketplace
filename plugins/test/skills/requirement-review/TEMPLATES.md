# 需求评审模板集

本文件包含 `requirement-review` 工作流各阶段使用的中间文件和输出模板。

**目录**（按阶段使用顺序）：

1. `requirement_understanding.md` 模板 — understand 阶段输出
2. `review_checklist.md` 模板 — review 阶段输出
3. `report.md` 模板 — output 阶段输出（含 7 章结构定义）
4. 问题分配规则 — review/output 阶段使用
5. 优先级定义 — review/output 阶段使用

---

### 各章节格式规则

**第 1 章「评审结论」**

- 第一行格式固定：`就绪度：{emoji} {判定}`，emoji 三选一：`✅`（就绪）/ `⚠️`（有条件就绪）/ `❌`（不就绪）
- 紧跟一段自然语言总结（1-2 句），概括核心问题域，不泛泛而谈
- 就绪条件列表：每行格式 `- A-N: 问题简述 | 所需行动`，**不写来源编号**（FB2 等内部编码对读者无意义）
- 元数据用粗体标题 + 列表呈现，不使用表格
- 元数据末尾必须包含 `分析方式：AI 辅助 + 评审会决策`，明确强调结论由人在评审会最终决策（不写「评审人：QA AI 助手」之类的署名，避免削弱可信度）

**第 2 章「需求理解」**

- 需求目标：以 `需求目标：` 开头，1-2 句话。不复述用户场景
- 功能点清单**必须**用 `<details><summary>...</summary>...</details>` 包裹默认折叠
  - 飞书云文档导入后会渲染为可展开 block，避免 46+ 条清单霸占报告 1/3 篇幅
- 在 details 内按模块 `### {模块名}` 分组，每个功能点独占一行 `- F{N}: {功能点描述} | 显式/隐式`
- 功能点必须细粒度（每个可独立测试），不能笼统分组

**第 3 章「各职能待确认问题列表」**

报告的核心价值章节。按 PM / Dev / QA / Design 分组。

格式规则：
- 每组以 `### {角色}（{中文名}）` 为标题
- 每个问题独占一行，格式 `- {角色}-{N}: {问题描述} | {优先级} | {维度名} | {建议确认时间}`
- **不写来源编号**（FB2、LC2 + UX3 等内部编码对人类读者无意义；维度名足够定位）
- 问题描述必须具体、可回答，包含足够上下文（如引用文档中的矛盾文案原文）
- 阻断项排在每组最前面

**第 4 章「时间盒清单」（report.md 必出）**

按建议处理时机重新组织 §3 + §5 的待办，方便各角色拿到自己时间点上的清单。

固定 4 个时间点：评审会前 / 评审会上 / 开发前 / 提测前。

格式规则：
- 每个时间点用 `**{时间点}**` 粗体小标题
- 每条 to-do 引用对应的问题/风险编号（如 `[PM-1, Dev-2]`），后面跟一句话动作描述
- 不重复展开问题原文（避免与 §3 / §5 内容大段冗余）
- 如果某时间点无待办，写 `- （无）`

**第 5 章「风险项摘要」**

风险筛选标准——满足以下任一条件的评审发现应提升为风险项：
- 性能要求维度中预估数据量级过大或响应时间有风险的项
- 异常处理维度中影响核心流程的项
- 依赖关系维度中存在外部不可控因素的项
- 影响范围维度中涉及跨模块或跨系统变更的项
- 时间依赖或环境依赖可能导致延期的项

格式：每行 `- R-{N}: {风险描述} | {优先级} | {影响范围} | {建议措施}`
无风险时标注 `- 未发现重大风险`。

**第 6 章「评审范围声明」**

按 `**评审文档**` / `**信息边界**` 两组粗体小标题组织：
- **评审文档**：列出实际读取的 markdown 文件列表（含主/子文档数量、附图数量）
- **信息边界**：每行用 emoji 标注获取状态
  - `✅` 已获取
  - `❌` 未获取
  - `⚠️` 文档自身标注未完成

> ⚠️ 旧版本曾有「第 7 章 评审完整性声明」（功能点覆盖率、阻断项比例 X%「在合理范围内」、问题发现分布等），现已**整节删除**。原因：90% 信息与 §1 元数据 / §3 章节末尾计数重复，且「阻断项 8.3% 在合理范围内」之类的 AI 自我评价对读者无价值。

### 关键约束

- 整份报告使用纯文本和分隔线排版，**不使用 markdown 表格**（飞书导入后表格渲染不稳定）
- 所有问题详情归入第 3 章
- 第 3 章每个问题独占一行，格式统一为 `- 编号: 问题 | 优先级 | 维度名 | 确认时间`
- 阻断项通常不超过总问题数的 20%；若超过，在第 1 章评审结论中说明需求成熟度不足
- 功能点编号（F1/F2...）在报告全文中保持一致，第 3 章问题可通过功能点编号交叉引用
- 状态符号统一使用 emoji：`✅` 通过 / `⚠️` 有条件 / `❌` 不通过；**禁止**使用 `[OK]` / `[!]` / `[X]` 等 ASCII 兜底符号

## 问题分配规则

将评审发现的问题按以下规则分配给对应职能。12 维度 → 职能映射：

| 评审维度 | 主分配 | 副分配 |
| --- | --- | --- |
| 功能边界 | PM | — |
| 状态流转 | PM | Dev |
| 性能要求 | Dev | — |
| 权限控制 | Dev | QA |
| 异常处理 | Dev | QA |
| 数据约束 | Dev | QA |
| 依赖关系 | Dev | PM |
| 交互与 UI 规则 | Design | PM |
| 安全与合规 | Dev | — |
| 兼容性 | Dev | QA |
| 可测试性与验收标准 | QA | PM |
| 影响范围 | PM | Dev |

补充分配规则（适用于跨维度问题）：

| 问题来源 | 分配给 |
| --- | --- |
| 跨端一致性、多端同步 | PM + Dev（联合确认） |
| 测试环境、测试数据、提测时间、Mock 方案 | QA（测试） |
| 空状态设计、加载状态、错误提示 | Design（设计） |
| 文档-设计稿差异（功能覆盖） | PM（产品） |
| 文档-设计稿差异（交互行为/UI 状态） | Design（设计） |
| 文档-设计稿差异（数据字段） | PM + Dev（联合确认） |

## 优先级定义

| 优先级 | 含义 | 建议确认时间 |
| --- | --- | --- |
| 阻断 | 不确认无法进入开发，严重影响需求范围 | 评审前或评审会上必须确认 |
| 高 | 影响核心功能或有较大风险 | 评审会上确认 |
| 中 | 影响部分功能或需要补充说明 | 开发前确认 |
| 低 | 优化建议或细节问题 | 开发过程中确认即可 |
