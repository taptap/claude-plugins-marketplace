# 需求分析 12 维度框架

本文件定义 `requirement-clarification` 和 `requirement-review` 共享的 **12 个功能点级维度**，以及 1 个**文档级附加项（PRD 文档质量校对）**。各 skill 的 CHECKLIST.md 引用本文件，仅补充 skill 特有的检查项和使用策略。

术语定义见 [CONVENTIONS.md](../../CONVENTIONS.md#术语表)。

## 何时选用哪个 Skill

| 场景 | 推荐 Skill | 产出 |
|---|---|---|
| 拿到新需求，需要通过问答拉齐理解 | requirement-clarification | `clarified_requirements.json`（供下游 skill 消费） |
| 需求文档已写好，评审会前质量把关 | requirement-review | `review_checklist.md` + `report.md`（供评审会使用） |
| "帮我理清需求" | requirement-clarification | — |
| "帮我评审需求" | requirement-review | — |
| 不确定用哪个 | 看目的：**拉齐理解**→clarification，**质量检查**→review | — |

## 1. 功能边界

**检查点**：
- 需求描述的功能范围是否有明确的内/外边界
- 是否存在模糊的「等」「等等」「类似」表述
- 新功能与现有功能的交互边界
- 功能是否覆盖完整 CRUD 链路（创建/查看/编辑/删除）

**问题模板**：
- 「{功能}的范围是否包含{子功能A}和{子功能B}？」
- 「{功能}是否仅限{条件}场景，还是所有场景通用？」
- 「{功能}与现有{模块}的关系是替换、扩展还是独立？」

## 2. 状态流转

**检查点**：
- 核心实体是否有完整的状态定义
- 状态间的转换规则是否明确（合法/非法迁移）
- 终态是否可逆
- 并发状态变更的处理规则
- 业务规则之间是否存在冲突
- 数据流转是否闭环

**问题模板**：
- 「{实体}有哪些状态？状态之间的转换条件是什么？」
- 「{实体}处于{状态A}时，能否直接转到{状态C}，还是必须经过{状态B}？」
- 「{实体}到达{终态}后是否可以回退？」
- 「两个操作同时修改{实体}状态时如何处理？」

## 3. 性能要求

**检查点**：
- 响应时间要求（页面加载、接口响应）
- 数据量级预估（列表上限、批量操作上限）
- 并发场景预估
- 分页/懒加载策略
- 是否需要缓存策略
- 性能风险评估（大数据量、高并发场景）

**问题模板**：
- 「{列表}最多展示多少条数据？是否需要分页？」
- 「{操作}的预期响应时间是多少？」
- 「{功能}是否需要支持批量操作？上限是多少？」

## 4. 权限控制

**检查点**：
- 涉及的用户角色及其权限差异
- 数据可见性规则（哪些角色能看到哪些数据）
- 操作权限（哪些角色能执行哪些操作）
- 未登录/游客的访问限制

**问题模板**：
- 「{功能}涉及哪些用户角色？各角色的权限差异是什么？」
- 「{数据}对{角色A}和{角色B}的可见范围是否不同？」
- 「未登录用户能否访问{功能}？」

## 5. 异常处理

**检查点**：
- 网络异常时的降级策略
- 操作失败后的回退/重试机制
- 数据一致性保障（如支付场景）
- 中断场景处理（页面刷新、切后台、超时）
- 回退/撤销方案是否考虑

**问题模板**：
- 「{操作}失败后，{实体}的状态如何处理？用户看到什么提示？」
- 「{操作}进行中用户刷新页面/切后台，回来后如何恢复？」
- 「{操作}超时后是自动重试还是需要用户手动操作？」

## 6. 数据约束

**检查点**：
- 输入字段的长度/格式/类型限制
- 唯一性约束（如用户名不可重复）
- 必填/选填规则
- 数据保留和清理策略
- 边界条件是否有定义（最大值、最小值、空值、特殊字符、超长文本）

**问题模板**：
- 「{字段}的长度限制是多少？是否允许特殊字符？」
- 「{字段}是否需要唯一？重复时如何提示？」
- 「{数据}的保留期限是多久？过期后如何处理？」

## 7. 依赖关系

**检查点**：
- 上下游系统依赖（第三方 API、内部服务）
- 多端一致性要求（Server/Android/iOS/Web/PC）
- 多端是否同步开发和上线？如果存在时间差，业务表现预期需要对齐
- **前后端 API 契约**（当 `platform_scope.coordination_needed` 为 true 且源材料包含 API 相关技术信息时执行。`api_evidence_level` 为 `"none"` 时跳过此子维度）：
  - 新增接口的 method、path、request/response 字段定义
  - 现有接口是否需要新增字段或修改行为
  - 错误码定义和异常响应格式
  - 接口版本兼容策略（新旧客户端共存期间）
- 是否需要 DE/IEM 支持（数据埋点）
- 是否涉及 TapSDK
- 发版依赖（是否依赖宿主发版）
- 并行项目冲突风险
- 是否需要新增数据表/字段
- 是否涉及数据迁移

**问题模板**：
- 「{功能}是否依赖{外部服务}？该服务不可用时如何降级？」
- 「多端（Android/iOS/Web）的行为是否完全一致？是否同步上线？」
- 「{功能}需要新增 API 接口吗？请求和响应的字段有哪些？」
- 「{现有接口}是否需要新增字段？旧版客户端如何兼容？」
- 「是否需要新增数据埋点？具体事件是什么？」
- 「是否有其他并行项目会改到同一块代码？」

## 8. 交互与 UI 规则

**检查点**：
- 用户操作流程和页面跳转路径
- 表单交互规则（实时校验/提交校验、联动、默认值）
- 加载状态和空状态展示
- 动效和手势交互（移动端）
- 提示信息（成功/失败/确认弹窗）
- 信息层级是否清晰
- 如有设计稿，对照设计稿验证交互逻辑和状态覆盖
- **多变体一致性**（条件触发 — 见下方）：当需求在某 cell / 列表 / tab 上**新增 UI 元素**（按钮、icon、menu）或**修改组件行为**时，必须主动追问该改动是否对所有 tab / 子页面 / 列表变体都适用，避免类继承或父组件级联导致的隐式扩散。**重要**：此问题独立于代码视角的影响面分析（`change-analysis`），是从产品语义视角的主动澄清，必跑
- **图标资源来源**（条件触发 — 仅当需求涉及 icon / 图标资源时，必跑）：必须先判断 icon 是**新增**还是**复用现成图标**，并据此走不同澄清路径：
  - **新增**：必须从设计稿获取 icon 资源 ID（如 iconfont code `.uexxxx` / Figma node id / asset 资源名）；设计稿未提供或找不到时，**必须** AskUserQuestion 向用户/设计师确认，**禁止** AI 自造资源、用 SVG 替代 iconfont、或将 Android XML drawable 转 SVG 后伪装成 imageset
  - **复用**：必须列出仓库现有候选 icon（含 icon code / 预览）让用户从中选择，**禁止** AI 直接挑一个填上去
  - **必须查仓库现有 icon 体系**再下结论：iOS iconfont（统一管理路径如 iconfont.cn 项目 ID）、Android drawable、Web SVG sprite — 大概率已有可复用资源
  - **遵循仓库 icon 管理惯例**：不同来源（imageset / iconfont / 远程图）不可互相替代，按仓库现有惯例选择正确的资源类型（如 iOS 统一用 `UIImage.iconFont(.uexxxx, fontSize: ...)` 而非 imageset）

**问题模板**：
- 「{操作}完成后跳转到哪个页面？是否有成功提示？」
- 「{表单}中{字段A}变化时，{字段B}是否联动更新？」
- 「{列表}为空时展示什么？加载中展示什么？」
- 「{操作}是否需要二次确认弹窗？」
- 「{列表项}支持左滑删除 / 长按拖拽排序吗？」
- **多变体一致性追问**（条件触发）：「{页面/列表}的所有 tab / 子页面 / 变体（如 收到的 / 发出的 / 提到我的）是否都需要 {新增的 UI 元素 / 行为}？是否有特定 tab 不应展示？」选项建议：A. 全部适用；B. 部分适用（请列出例外）；C. 不确定（建议 PM 与开发对齐每个变体的预期行为）
- **图标资源来源追问**（条件触发）：
  - 「{页面/组件}用到的 {icon 名} 是**新增图标**还是**复用现成图标**？」选项：A. 新增（请提供设计稿的 icon 资源 ID）；B. 复用现有（请告知具体是哪个 icon）；C. 不确定（建议与设计师对齐）
  - 新增分支：「该图标的设计稿资源 ID 是什么？（iconfont code / Figma node / asset 资源名）」— 找不到时必须追问，禁止 AI 自造或用 SVG 替代
  - 复用分支：「我在仓库中查到以下候选：{icon-A code / icon-B code / icon-C code}，请确认使用哪一个？」— 没查到候选时直接说明并请用户提供，禁止凭印象选一个

## 9. 安全与合规

**条件触发**：仅当需求涉及用户输入、权限、支付或敏感数据时逐项检查，否则整体标记为 `[不适用]`。

**检查点**：
- 新增/修改的用户输入是否有校验方案（防 XSS、注入）
- 权限控制变更是否考虑越权场景（水平越权、垂直越权）
- 涉及金额计算时精度和防篡改方案是否明确
- 敏感数据的存储/传输/展示是否有脱敏方案
- 日志中的敏感信息脱敏规则
- 操作审计记录需求
- 隐私法规合规（如需用户授权才能采集的数据）
- 防刷和风控策略

**问题模板**：
- 「{敏感数据}在日志和展示中是否需要脱敏？脱敏规则是什么？」
- 「{操作}是否需要记录审计日志？记录哪些字段？」
- 「{功能}涉及的用户数据采集是否需要用户授权？」
- 「{接口}是否需要防刷限制？频率是多少？」

## 10. 兼容性

**检查点**：
- 浏览器版本支持范围（Web 端）
- 操作系统最低版本要求（移动端）
- 屏幕尺寸适配策略（响应式/自适应/固定宽度）
- 旧版本客户端的降级或强制升级策略
- 新老版本兼容方案如何设计
- 国际化/本地化需求

**问题模板**：
- 「需要支持哪些浏览器？最低版本是什么？」
- 「最低支持的 Android/iOS 版本是多少？」
- 「小屏幕（如 320px 宽）下{组件}如何展示？」
- 「旧版本客户端访问新功能时如何处理？强制升级还是降级展示？」
- 「是否需要支持多语言？首批支持哪些语言？」

## 11. 可测试性与验收标准

**检查点**：
- 每个功能点的验收标准是否明确且可验证
- 是否需要特殊的测试数据或测试账号
- 是否有需要模拟的外部依赖（Mock 服务）
- 验收由谁执行、在什么环境执行
- 是否需要特殊测试手段（自动化/性能/安全/兼容性测试）
- 灰度/AB 测试需求
- 变更影响的回归范围是否可评估

**问题模板**：
- 「{功能点}的验收标准是什么？如何判断"做完了"？」
- 「测试{功能}需要准备哪些测试数据？是否需要特殊账号？」
- 「{外部依赖}在测试环境中如何处理？有 Mock 服务吗？」
- 「{功能}是否需要灰度发布？灰度策略是什么？」

## 12. 影响范围

**检查点**：
- 变更涉及的模块和功能（直接影响 + 间接影响）
- 上下游系统是否需要同步调整
- 数据库字段变更是否影响报表、数据分析
- 是否需要数据迁移或回填
- 共用组件/SDK 的修改是否影响其他业务
- 是否影响现有接口（向前/向后兼容）
- 是否有技术债务风险

**问题模板**：
- 「修改{模块}会影响哪些下游模块？{列表}是否都在本次范围内？」
- 「{字段}变更后，报表/数据分析是否需要同步调整？」
- 「{共用组件}的修改是否会影响其他业务线？」
- 「是否需要数据迁移？历史数据如何处理？」

## 附加项：PRD 文档质量校对

> 本节是对 PRD **文档文本本身**的质量检查（与上面 12 个功能点级维度并列，但属于文档级元检查）。`requirement-review` 在 12 维度评审完成后**必须**执行；`requirement-clarification` 在功能点 × 维度矩阵分析完成后**必须**执行。每条发现作为单独区块写入产物（review 写入 `review_checklist.md` 的「文档文案校对」区块；clarification 写入 `clarified_requirements.json` 的 `doc_quality_issues` 字段）。

**为什么单独拎出来**：错别字、术语漂移会原样传染到端上文案、UI 标签和测试用例（真实案例：PRD 写「该贴」→ Android/iOS 复制为「该贴」→ 上线后才发现应为「帖」）。AI 阅读 PRD 时只关注语义不易察觉这类问题，需要显式触发一遍校对。

**严重性术语映射**：本节正文使用中文 `阻断 / 关注`；写入 JSON 字段（如 `doc_quality_issues[].severity`）时映射为 `blocking / concern`。两套术语等价，仅渲染层（人读 vs 机读）不同。

**检查点**：

1. **错别字与易混汉字**：常见混淆字 — 贴/帖、的/得/地、登录/登陆、做/作、账/帐、再/在、按耐/按捺、即/既、效/笑（音近形近）。命中时引用 PRD 原文位置 + 建议改写。
2. **术语一致性**：同一概念在 PRD 中是否始终使用同一称谓（例：「帖子 / 内容贴 / post / 动态」混用）；与公司术语表（CONVENTIONS.md 术语表、TapSDK / 建站器 / DE / IEM 等）的写法是否一致。
3. **易读性 / 流畅性**：超长句（>80 字且含 ≥3 个分句且无标点切分）、指代不清（"它/该/此"无法唯一回溯到名词）、被动堆叠、双重否定、连续口语化连接词（"然后…然后…然后"）。
4. **文案与 UI / 设计稿一致性**：PRD 中引述的按钮文案、toast 文案、错误提示文案与设计稿/截图中的实际字串是否完全一致（含标点、空格、大小写、全/半角）。如设计稿不可用则跳过此项并标注 `[不适用：无设计稿]`。
5. **数字、单位、占位符一致性**：金额单位（元/分）、时间单位（秒/毫秒）、阈值（>= 还是 >）、占位符格式（`{name}` vs `${name}`）在全文是否统一。

**输出格式**（每条问题独立一行）：

```
- [错别字] 第 X 节『原文片段』→ 建议：『修改后片段』
- [术语不一致] 全文「A」「B」「C」混用指同一概念 → 建议统一为「A」
- [易读性] 第 X 节『原文长句』→ 建议拆分为两句：…
- [文案不一致] PRD「按钮文案 X」与设计稿『Y』不一致 → 与 PM/Design 确认以哪个为准
```

**严格基线**：每条发现必须含**原文摘录**（用 `『』` 圈出），禁止仅给结论不给原文；找不到原文 = 不写这条（防幻觉）。建议改写若不确定则只指出问题、注明「建议 PM 校对确认」，不强行给出新文案。

**判定为阻断/关注**：
- 错别字若已扩散到 UI 文案 / 端实现引用（如设计稿、Android 已照抄）→ 阻断（必须在评审会上确认是否回滚）
- 术语不一致 / 易读性问题 → 关注项（开发前 PM 校对即可）
- 数字单位歧义 / 占位符不一致 → 阻断（直接影响实现）
