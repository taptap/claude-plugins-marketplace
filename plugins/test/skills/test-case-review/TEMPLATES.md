# 用例评审模板集

本文件定义 `test-case-review` 工作流 `summary` 阶段产出的 `review_summary.md`（最终上传到飞书云文档的人类可读报告）格式。

> **关于符号约定**：本 skill 输出报告（review_summary.md）的状态标记使用中文方括号 `[已覆盖]/[部分覆盖]/[未覆盖]`、`[通过]/[有条件]/[不通过]`，**禁止使用 emoji ✅/⚠️/❌**——飞书 import 会把 ✅→`[OK]`、⚠️→`[!]`、❌触发 bitable block 解析导致整行内容丢失（详见 §「关键约束」）。

---

## review_summary.md 模板

> `summary` 阶段输出。这是最终上传到飞书云文档的报告，必须严格按本模板组织。

### 完整报告示例骨架

> 设计原则：飞书云文档名已经是「{需求名称}-用例评审-{YYYYMMDDHH}」，故 review_summary.md **不写 H1 标题**，避免飞书导入后出现重复标题。

```markdown
## 0. 评审结论

用例就绪度：[有条件] 有条件通过
存在 N 个 P0 阻塞项需立即修复，M 个 P1 改进项建议本轮跟进。核心问题集中在……（一句话概括问题域）。

待修复（按优先级）：
- P0 阻塞 N 个：{一句话总结，如「3 个用例端到端不闭环 + 1 个核心需求未覆盖」}
- P1 改进 M 个：{一句话总结}
- P2 优化 K 个：{一句话总结}

**元数据**
- 需求名称：XXX
- 需求链接：https://...
- 评审日期：YYYY-MM-DD
- 评审范围：{用例总数} 条 × {评审维度个数} 维度
- 分析方式：AI 辅助 + 评审人决策

## 1. 评审总览

**核心指标**
- 需求功能点：N 个（RP-1 ~ RP-N）
- 测试用例：M 条（P0: a / P1: b / P2: c / P3: d）
- 需求覆盖率：x/N 完整覆盖（X%），y/N 部分覆盖，z/N 未覆盖（**目标 ≥ 90%**）
- 场景缺失：K 个（P1: a / P2: b / P3: c）
- 正确性问题：J 处
- 规范性问题：L 处

## 2. 需求覆盖率

### 2.1 已覆盖（x/N）
- RP-1 {需求点描述} → 用例 {用例ID列表}
- RP-2 {需求点描述} → 用例 {用例ID列表}
- ...

### 2.2 部分覆盖（y/N）
- RP-K {需求点描述} → 用例 {用例ID列表}
  - 缺什么：{具体说明 + 建议补充什么}
- ...

### 2.3 未覆盖（z/N）
- RP-M {需求点描述} → **建议新增用例**：{用例方向}
- ...

## 3. 场景完整性

### 3.1 端到端链路不闭环
- 用例 {用例ID}（{用例名}）：只有 N 个步骤，缺少 ①{X}；②{Y}；③{Z}
- ...

### 3.2 缺失边界场景
按"关联需求点 / 缺失场景描述 / 建议优先级"组织（不使用 markdown 表格）：

**RP-7 相关**
- [P1] 白名单模式只选启动器包未选游戏本体包时尝试提交的系统行为
- [P2] 更新日志超过 5000 字符截断/提示

**RP-10 相关**
- [P2] 白名单→非白名单切换时若有审核中包体的处理

## 4. 用例正确性

### 4.1 预期结果有误/不完整
- {用例ID}：当前预期"{原预期}"——{问题}。修改建议：{建议}
- ...

### 4.2 步骤术语错误
- {用例ID}（{用例名}）步骤N：{问题描述} + 修改建议
- ...

### 4.3 优先级合理性
- {用例ID}（{用例名}）当前 P3，建议调整为 P2。理由：{业务含义}
- ...

## 5. 用例规范性

### 5.1 命名问题
- {用例ID}（当前名："{原名}"）：{问题} → 建议改为"{新名}"

### 5.2 步骤规范问题
- {用例ID}：{问题描述} → 建议拆分为步骤①{X}；②{Y}

### 5.3 前置条件规范问题
- {用例ID}：{问题描述} → 建议补充"{补充内容}"

## 6. 修复 to-do（按用例 ID 分组）

> 同一个用例可能在 §3 / §4 / §5 多处被点名，本节按用例 ID 一次列全所有需要修的问题，方便修复者直接拿走。

**P0 阻塞**

- **用例 {用例ID}（{用例名}）**
- · [正确性] §4.1.1 — 步骤拆分 + 预期版本号验证
- · [规范性] §5.2.1 — 模拟低版本说明

- **用例 {用例ID2}（{用例名}）**
- · [完整性] §3.1.1 — 端到端步骤补全

**P1 改进**

- **用例 {用例ID3}（{用例名}）**
- · [正确性] §4.2.1 — 步骤术语"游戏整包"应为"游戏包"

**P2 优化**

- **用例 {用例ID4}（{用例名}）**
- · [规范性] §5.1.1 — 名称含问号，改为陈述句

## 7. 建议补充的用例（本次跳过创建）

> 如已开启自动生成，对应用例已写入 `supplementary_cases.json`，本节仅做清单展示。

**P1**
- {用例方向} — 关联需求点 RP-N

**P2**
- {用例方向} — 关联需求点 RP-N
```

### 各章节格式规则

**第 0 章「评审结论」**

- 第一行格式固定：`用例就绪度：{标记} {判定}`，标记三选一：`[通过]` / `[有条件]` / `[不通过]`
  - **禁止使用 emoji** ✅/⚠️/❌ 与 ASCII `[OK]/[!]/[X]`：实测飞书 import 转换问题
- 紧跟一段自然语言总结（1-2 句），概括核心问题域
- 待修复列表按 P0/P1/P2 分级，每级一句话总结，**不展开问题详情**（详情在 §6）
- 元数据末尾必须包含 `分析方式：AI 辅助 + 评审人决策`，不写「评审人：QA AI 助手」之类的署名

**第 1 章「评审总览」**

- 用粗体小标题 + 列表，**不使用 markdown 表格**（飞书 import 散行问题）
- 必须包含覆盖率 + 目标值（默认 ≥ 90%；项目可在 PHASES.md 调整目标线）
- 用例数按优先级 P0/P1/P2/P3 分布
- 问题数按 4 个维度分别给出（覆盖/完整性/正确性/规范性）

**第 2 章「需求覆盖率」**

按"已覆盖 / 部分覆盖 / 未覆盖" 三组分别列出（**不使用 markdown 表格**）：
- §2.1 已覆盖：每行 `- RP-N {描述} → 用例 {ID列表}`，简洁一行
- §2.2 部分覆盖：每行先列 `- RP-N {描述} → 用例 {ID列表}`，下一行缩进 `- 缺什么：{说明}`
- §2.3 未覆盖：每行直接给出补充建议方向

**第 3 章「场景完整性」**

- §3.1 端到端不闭环：列具体用例 ID，描述缺什么步骤
- §3.2 缺失边界场景：按"关联需求点"分组，每条用 `[P1] / [P2] / [P3]` 前缀标优先级（不用表格）

**第 4 章「用例正确性」**

按 4.1 / 4.2 / 4.3 三个子维度组织，每条问题独占一行 `- {用例ID}：{描述} → 修改建议`。

**第 5 章「用例规范性」**

按 5.1 命名 / 5.2 步骤 / 5.3 前置条件 三个子维度组织。

**第 6 章「修复 to-do（按用例 ID 分组）」**

- 报告的核心可执行价值章节
- 同一个用例的所有问题（跨 §3 / §4 / §5）汇总到一起
- 按 P0 / P1 / P2 分级
- 用例 ID 行用 **粗体 + 用例名**：`- **用例 {用例ID}（{用例名}）**`
- 待办行用 `- · ` 前缀（点号视觉缩进）+ `[维度] §章节编号 — 一句话动作`
  - **不使用嵌套 markdown bullet**：飞书 import 会拍平嵌套 bullet 层次，导致用例 ID 行与待办行视觉上同级
  - 用 `· ` 字符 + 粗体差异让分组层次保留可读性
- 修复者直接 grep 自己负责的用例 ID 即可拿到全部待办

**第 7 章「建议补充的用例」**

- 与 §2.3 未覆盖呼应，但更具体（用例方向描述）
- 按 P1 / P2 分组（P0 通常已在 supplementary_cases.json 自动生成）
- 标注关联的 RP-N

### 关键约束

- 整份报告使用纯文本和分隔线排版，**不使用 markdown 表格**（飞书 import 后表格被拆成离散 text block，看起来像散文）
- 不写 H1 标题（飞书 doc 名已包含「{需求名称}-用例评审-{YYYYMMDDHH}」）
- **状态符号统一使用中文方括号标记**（非 emoji 非 ASCII）：
  - 用例就绪度：`[通过]` / `[有条件]` / `[不通过]`
  - 覆盖状态：`[已覆盖]` / `[部分覆盖]` / `[未覆盖]`
  - 优先级：`[P0]` / `[P1]` / `[P2]` / `[P3]`
- **禁止使用 emoji** ✅/⚠️/❌：实测飞书 import 会把 ✅ 转 `[OK]`、⚠️ 转 `[!]`、❌ 触发 bitable block 解析导致内容丢失
- **禁止 §6 复述 §3/§4/§5 的问题详情** —— §6 是按用例 ID 重新组织的索引，子条目只引用章节编号 + 一句话动作，避免大段重复
- **不写「评审人：QA AI 助手」**之类的 AI 署名 —— 用「分析方式：AI 辅助 + 评审人决策」强调结论由人在评审会最终决策
- 用例 ID 在报告全文中保持一致（hex hash 或其他业务 ID），跨章节可通过 ID 交叉引用
